3. API ROUTES (Backend Endpoints)
================================================================================

---------- INTERVIEW ROUTES ----------

3.1  POST /api/interview/generate
  File: app/api/interview/generate/route.ts
  Purpose: Generate an interview question (alternative simpler endpoint)
  Input (JSON body):
    {
      systemPrompt: string,    // Required. Full system prompt
      messages: Array<{role: 'user'|'assistant', content: string}>,  // Required
      config?: { temperature?: number, maxTokens?: number }
    }
  Output (JSON):
    Success: { ok: true, data: RecruiterResponse }
    Error:   { ok: false, error: { code, message, hint } }
  LLM Client Function: nextTurn()
  Error Codes: INVALID_REQUEST, LLM error codes, UNKNOWN_ERROR

3.2  POST /api/interview/next-turn
  File: app/api/interview/next-turn/route.ts
  Purpose: Main interview turn handler (initializes + processes Q&A turns)
  Input (JSON body, validated with Zod):
    {
      sessionId: string,                         // Required
      isInit: boolean,                            // Required  
      interviewParams?: {                         // Required if isInit=true
        role: string,
        level: string,
        interviewType: string,
        language: string,
        duration?: number
      },
      candidateText?: string,                     // Required if isInit=false
      messages: Array<{role: 'user'|'assistant', content: string}>
    }
  Output (JSON):
    Success: {
      ok: true,
      data: {
        say: string,
        type: "question"|"followup"|"closing",
        rubric: "hr"|"tech"|"closing",
        evaluation?: {
          total_score: number,
          technical_score: number,
          communication_score: number,
          problem_solving_score: number,
          experience_score: number,
          signals?: string[]
        }
      },
      meta: { requestId, latency, llmLatency }
    }
    Error: { ok: false, error: { code, message, hint, requestId }, meta? }
  Special Behavior:
    - Maintains in-memory session cache (Map<sessionId, sessionData>)
    - On init: builds system prompt from interviewParams, sends to LLM
    - On turn: appends candidateText to history, sends full context to LLM
    - Validates LLM response with Zod schema
    - Returns 200 with ok:false for expected LLM errors (not 500)
    - Exports cleanupSessions() (unused)
  LLM Client Function: nextTurn()
  Error Codes: INVALID_JSON, VALIDATION_ERROR, MISSING_PARAMS, MISSING_TEXT,
               INVALID_MODEL_JSON, LLM codes, INTERNAL_ERROR

3.3  POST /api/interview/summary
  File: app/api/interview/summary/route.ts
  Purpose: Generate final interview evaluation summary
  Input (JSON body):
    {
      summaryPrompt: string    // Required. Full summary prompt text
    }
  Output (JSON):
    Success: { ok: true, data: InterviewSummary }
    Error:   { ok: false, error: { code, message, hint } }
  LLM Client Function: generateSummary()

  Purpose: Test LLM API connectivity and measure latency
  Input: None
  Output (JSON):

    Error:   { ok: false, error: { code, message, hint }, latency: number }

---------- SESSION ROUTES ----------

3.5  POST /api/sessions/create
  File: app/api/sessions/create/route.ts
  Purpose: Create a new interview session record in PostgreSQL
  Input (JSON body):
    {
      role: string,            // Required
      level: string,           // Required
      interviewType: string,   // Required
      language: string,        // Required
      durationMinutes?: number,
      duration?: number        // fallback for durationMinutes
    }
  Output (JSON):
    Success: { ok: true, sessionId: string }
    Error:   { ok: false, error: { code, message, requestId } }
  Database: prisma.interviewSession.create()
  Error Codes: VALIDATION_ERROR, DB_UNAVAILABLE (P1001/P1002), SERVER_ERROR

3.6  GET /api/sessions/list
  File: app/api/sessions/list/route.ts
  Purpose: List all interview sessions (sorted by createdAt desc)
  Input: None
  Output (JSON):
    Success: {
      ok: true,
      sessions: Array<{
        id, createdAt, endedAt, role, level, interviewType,
        status, recruiterImpression, overallScore
      }>
    }
    Error: { ok: false, error: { code, message, requestId } }
  Database: prisma.interviewSession.findMany()

3.7  GET /api/sessions/[id]
  File: app/api/sessions/[id]/route.ts
  Purpose: Get a single session with messages and final report
  Input: URL parameter: id (session UUID)
  Output (JSON):
    Success: {
      ok: true,
      session: {
        ...InterviewSession,
        messages: Array<InterviewMessage> (timestampMs converted to Number),
        finalReport?: InterviewFinalReport
      }
    }
    Error: { ok: false, error: { code: 'NOT_FOUND'|'DB_UNAVAILABLE'|'SERVER_ERROR' } }
  Database: prisma.interviewSession.findUnique() with include: { messages, finalReport }

3.8  POST /api/sessions/[id]/message
  File: app/api/sessions/[id]/message/route.ts
  Purpose: Save a single interview message to the database
  Input (JSON body):
    {
      role: string,            // Required ('recruiter' | 'candidate')
      text: string,            // Required
      timestampMs: number,     // Required (BigInt in DB)
      elapsedSec?: number,     // Defaults to 0
      recruiterMsgId?: string
    }
  Output (JSON):
    Success: { ok: true }
    Error: { ok: false, error: { code, message, requestId } }
  Database: prisma.interviewMessage.create()
  Special Behavior:
    - Returns ok:true even if DB is unavailable (graceful degradation)
    - Does NOT block the UI if persistence fails

3.9  POST /api/sessions/[id]/end
  File: app/api/sessions/[id]/end/route.ts
  Purpose: End interview session, generate AI final report, save to DB
  Input (JSON body):
    {
      messages: Array<{type: 'recruiter'|'user', text: string}>
    }
  Processing:
    1. Fetch session config from DB (role, level, etc.)
    2. Build conversation context from messages
    3. Build final evaluation prompt
    4. Call LLM for final evaluation (generateFinalReport())
    5. Update session status to 'ended', save scores
    6. Upsert InterviewFinalReport with impression, metrics, weaknesses
  Output (JSON):
    Success: {
      ok: true,
      finalReport: {
        recruiter_impression: 'Hire'|'Lean Hire'|'No Hire',
        metrics: {
          total_score: 0-100,
          technical_score: 0-100,
          communication_score: 0-100,
          problem_solving_score: 0-100
        },
        weaknesses: string[]
      }
    }
    Error: { ok: false, error: { code, message, requestId } }
  LLM Client Function: generateFinalReport()

  Input: None

5. STATE MANAGEMENT (Zustand Stores)
================================================================================

5.1  useAuthStore (store/auth.ts)
  Persistence: localStorage key "auth-storage"
  State:
    user: User | null              // { id, email, name, createdAt }
    isAuthenticated: boolean
  Actions:
    login(email, password)         → Fake auth (500ms delay), sets user
    register(email, password, name) → Fake auth, sets user
    logout()                       → Clears user
  MIGRATION NOTE: This is entirely fake. In Spring Boot, replace with real
  JWT/session authentication. The User model has { id, email, name, createdAt }.

5.2  useInterviewStore (store/interview.ts)
  Persistence: NONE (runtime only, resets on page reload)
  State:
    sessionId: string | null
    config: SessionConfig | null                    // { role, level, interviewType, language, duration }
    status: 'idle'|'connecting'|'interviewing'|'paused'|'ended'
    recruiterState: 'idle'|'listening'|'thinking'|'speaking'
    connectionStatus: 'disconnected'|'connecting'|'connected'|'reconnecting'
    recordingState: 'inactive'|'recording'|'paused'
    messages: ConversationMessage[]                 // { id, type, text, timestamp }
    partialTranscript: string
    liveInterim: string
    elapsedTime: number (seconds)
    audioLevel: number (0-100)
    privateNotes: PrivateNote[]                     // { timestamp, signals[], score_hint }
    isListening: boolean
    streamingText: string
    hasInitialized: boolean
    lastSentTranscript: string
    lastSentTimestamp: number
    isRecruiterSpeaking: boolean
    utteranceBuffer: string
    silenceTimerId: NodeJS.Timeout | null
    requestInFlight: boolean
    blockedUntil: number (timestamp)
    lastSentNormalized: string
    requestCounter: number
    preloadedResponse: { say: string, evaluation?: any } | null
    demoMode: boolean
  Actions: set* for each field, addMessage, appendStreamingText, reset(), etc.

5.3  useLLMStore (store/llm.ts)
  Persistence: localStorage key "llm-settings"
  State:
    provider: 'groq'
    baseUrl: 'https://api.groq.com/openai/v1'
    model: 'llama-3.3-70b-versatile'
    temperature: 0.4
    maxTokens: 1000
    useMockMode: boolean
  Actions: setProvider, setBaseUrl, setModel, setTemperature, setMaxTokens,
           setUseMockMode, getConfig()
  MIGRATION NOTE: These are client-side settings. In Spring Boot, move to
  application.yml or admin configuration.

5.4  usePreferencesStore (store/preferences.ts)
  Persistence: localStorage key "preferences-storage"
  State:
    language: 'EN' | 'FR'
    ttsEnabled: boolean (default true)
    autoSubtitles: boolean (default true)
    showGazeTips: boolean (default true)
    selectedAudioInput?: string (device ID)
    selectedVideoInput?: string (device ID)
  Actions: set* for each field

5.5  useTTSStore (store/ttsStore.ts)
  Persistence: localStorage key "tts-storage" (partial: only prefs, not runtime)
  State (persisted):
    enabled: boolean (default true)
    voiceURI: string | null
    rate: number (0.5-2.0, default 1.0)
    pitch: number (0.5-2.0, default 1.0)
    volume: number (0-1, default 1.0)
  State (runtime):
    isSpeaking: boolean
    lastSpokenMessageId: string | null

STATE FLOW:
  1. User configures interview on /onboarding → writes to interviewStore.config + preferencesStore
  2. User navigates to /interview → PreparationOverlay creates session, preloads AI
  3. Interview loop:
     a. Speech recognition → interviewStore (liveInterim, utteranceBuffer)
     b. Silence detection → API call → response → interviewStore (messages, streamingText)
     c. TTS speaks response → ttsStore (isSpeaking)
  4. Interview ends → API call → navigate to /session/[id]

9. ARCHITECTURE & DATA FLOW
================================================================================

OVERALL ARCHITECTURE:
  Client (React SPA) ←→ Next.js API Routes ←→ PostgreSQL
                                   ↕
                             Groq LLM API

DATA FLOW — INTERVIEW SESSION LIFECYCLE:

  1. SETUP (/onboarding)
     User selects: role, level, type, language, duration
     → Stored in useInterviewStore.config (Zustand, client memory)
     → Navigate to /interview

  2. PREPARATION (PreparationOverlay)
     a. POST /api/sessions/create { role, level, interviewType, language }
        → Creates InterviewSession in PostgreSQL (status: "running")
        → Returns sessionId → stored in useInterviewStore
     b. POST /api/interview/next-turn { sessionId, isInit: true, interviewParams, messages:[] }
        → Server builds system prompt from config
        → Calls Groq API with system prompt + "Start the interview" user message
        → Returns first recruiter question
        → Stored in useInterviewStore.preloadedResponse
     c. If API fails → fallback to demo mode (hardcoded responses)

  3. INTERVIEW LOOP (/interview page)
     a. PreloadedResponse displayed as first recruiter message
     b. TTS speaks recruiter message (via TTSQueue → Web Speech Synthesis)
     c. User speaks → Web Speech Recognition → interim text → final transcript
     d. Utterance buffering (concatenate finals until 2s silence)
     e. POST /api/interview/next-turn { sessionId, isInit:false, candidateText, messages }
        → Server appends candidateText to conversation history
        → Builds system prompt from cached session params
        → Calls Groq API with full conversation context
        → Validates response (Zod schema)
        → Returns RecruiterResponse { say, type, rubric, evaluation? }
     f. RecruiterResponse displayed → TTS speaks → recruiter avatar animates
     g. Messages saved:
        - POST /api/sessions/[id]/message (user message)
        - POST /api/sessions/[id]/message (recruiter message)
        (batched via persistence queue, flushes every 2s or 5 messages)
     h. Repeat from (c) until user clicks "End Interview"

  4. END INTERVIEW
     a. FlushAll pending messages to DB
     b. POST /api/sessions/[id]/end { messages: fullTranscript }
        → Fetches session config from DB
        → Builds final evaluation prompt
        → Calls Groq API (generateFinalReport)
        → Returns: impression, metrics (total/tech/comm/problem_solving), weaknesses
        → Updates InterviewSession: status='ended', endedAt, scores
        → Upserts InterviewFinalReport with all evaluation data
        → Returns finalReport to client
     c. Navigate to /session/[sessionId]

  5. REVIEW (/session/[id])
     a. GET /api/sessions/[id] → returns session + messages + finalReport
     b. Displays: transcript, scores, feedback, impression, weaknesses

  6. DASHBOARD (/dashboard)
     a. GET /api/sessions/list → all sessions (summary fields)
     b. Client computes: totalSessions, avgScore, streak, recentSessions


SERVER-SIDE REQUEST FLOW (for /api/interview/next-turn):
  Request → JSON parse → Zod validation → Session cache lookup/create
  → Build system prompt → Call Groq API (with rate limiting, retry, fallback)
  → Validate response schema → Update session cache → Return response

10. SPECIAL PATTERNS & MIGRATION NOTES
================================================================================

10.1  FAKE AUTHENTICATION
  - Current auth is entirely client-side (Zustand + localStorage)
  - No JWT, no sessions, no server-side auth
  - Anyone can access all pages, all API routes are unauthenticated
  MIGRATION: Implement Spring Security with JWT authentication.
  Add User entity to DB. Protect API endpoints. Associate sessions with users.

10.2  IN-MEMORY SESSION CACHE (next-turn route)
  - The /api/interview/next-turn route maintains an in-memory Map<sessionId, data>
  - This stores conversation history, interview params, and init status
  - VOLATILE: lost on server restart, not shared across instances
  MIGRATION: In Spring Boot, either:
    a. Use Redis for session cache (recommended for scalability)
    b. Use a stateful @Service with ConcurrentHashMap (single instance only)
    c. Reconstruct state from DB on each request (highest reliability)

10.3  DATABASE COMMUNICATION
  - Prisma ORM (Node.js) → replace with Spring Data JPA + Hibernate
  - UUID primary keys → use UUID type in PostgreSQL, @GeneratedValue(strategy=UUID)
  - BigInt timestampMs → use Long in Java
  - Json columns (signals, notesPrivate, whatIDidWell, areasForImprovement) →
    Use @Type(JsonType.class) with Hibernate Types or String with JSON parsing
  - Cascade deletes → JPA cascade types

10.4  LLM CLIENT MIGRATION
  - Current: Raw HTTPS requests using Node.js https module
  - Spring Boot equivalent: Use WebClient (reactive) or RestTemplate
  - Keep: Retry logic (3 retries, exponential backoff), model fallback,
    rate limiting (500ms gap), custom error codes
  - Create: GroqService @Service, GroqProperties @ConfigurationProperties
  - The API follows OpenAI spec → can use openai-java SDK

10.5  PROMPT ENGINEERING
  - Prompts are hardcoded in lib/prompts.ts
  - Three prompt types: interview system, summary, final evaluation
  - All require JSON-only output from the LLM
  MIGRATION: Create a PromptService or use template files (Thymeleaf/Mustache).
  Externalize prompt templates for easier modification.

10.6  CLIENT-SIDE BROWSER APIs (NOT IN SCOPE FOR SPRING BOOT BACKEND)
  These remain in the frontend (React SPA):
  - Web Speech Recognition (speech.ts)
  - Web Speech Synthesis / TTS (tts.ts, ttsQueue.ts)
  - MediaDevices / getUserMedia (media.ts)
  - MediaRecorder (media.ts)
  The Spring Boot backend only needs to expose REST APIs. The SPA handles these.

10.7  GRACEFUL DEGRADATION
  - /api/sessions/[id]/message returns ok:true even if DB is down
  - PreparationOverlay falls back to demo mode if AI API fails
  - LLM client uses fallback model if primary is deprecated/unavailable
  MIGRATION: Maintain this pattern in Spring Boot with @ExceptionHandler
  and @ControllerAdvice.

10.8  ENVIRONMENT VARIABLES
  Required for Spring Boot (application.yml):
    groq.api-key       (from GROQ_API_KEY)
    groq.model          (from GROQ_MODEL, default: llama-3.3-70b-versatile)
    spring.datasource.url   (from DATABASE_URL)
    spring.datasource.username
    spring.datasource.password

10.9  DOCKER
  - Current docker-compose.yml provides PostgreSQL 16 + Adminer
  - Can be reused for Spring Boot (just remove/replace the Next.js service)
  - Add a Spring Boot service with Dockerfile

10.10  ZOD VALIDATION → SPRING VALIDATION
  - Current: Zod schemas validate API request bodies
  - Migration: Use @Valid + Jakarta Bean Validation annotations
